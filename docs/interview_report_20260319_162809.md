## 二、参考答案（本场涉及）

### <a id="a-document-manager"></a>Q: 删除一个文档需要操作哪几个存储？失败怎么办？

必须协调删除四路存储：
1. Chroma 向量库
2. BM25 倒排索引
3. ImageStorage
4. `ingestion_history.db`

否则会出现 Hybrid Search 召回不一致（例如 BM25 召回到已在向量库删除的数据）。失败策略建议为 best-effort + 错误记录，生产可扩展到补偿/事务化删除。

### <a id="a-评估体系"></a>Q: Hit Rate@K 和 MRR 怎么计算？Ragas Faithfulness 衡量什么？

- Hit Rate@K：Top-K 内是否至少命中一条正确答案
- MRR：第一条正确结果排名倒数均值
- Faithfulness：回答是否基于检索上下文，是否有幻觉

### <a id="a-rrf公式"></a>Q: RRF 融合公式是什么？k 值含义？为什么不用线性加权？

$$
Score_{RRF}(d)=\frac{1}{k+Rank_{Dense}(d)}+\frac{1}{k+Rank_{Sparse}(d)}
$$

- `k` 是平滑因子，不是“防止 rank 为 0”这么简单
- RRF 用排名而非原始分值融合，避免 Dense 与 BM25 分值量纲不一致
- 线性加权通常需要归一化且对权重敏感，鲁棒性弱于 RRF

## 三、简历包装点评

### 包装合理 ✅
- “构建混合检索架构（Dense+BM25+RRF）”：能说出核心动机（分值不在同一尺度）与融合流程。
- “Prompt 迭代优化”：能给出从 v1 到 v2 的改动方向（先关键词再摘要）。

### 露馅点 ❌
- “全链路可观测性与自动化评估（量化可靠）” → 对评估标准的结构化定义不完整；Q6 中“4 个指标”仅明确 3 类，且阈值与一致性统计方法不严谨。**严重性：中**。
- “可支撑工程级多租户” → Q7/Q8 基本只覆盖向量 metadata，未覆盖 BM25 隔离、去重键租户化、MCP 入参鉴权校验。**严重性：高**。
- “RRF 机制掌握” → 公式表达成 `1/k+rank`，对 `k` 作用解释偏差。**严重性：中**。

### 改进建议
- 固化多租户设计清单：`tenant_id` 进入向量 metadata、BM25 索引命名空间、`ingestion_history` 复合唯一键（`tenant_id + file_hash`）、MCP tool 强制校验租户上下文。
- 评估体系补全：明确定义覆盖率/事实一致性/遗漏惩罚打分规则，加入双标注与 Kappa 统计。
- 背熟 RRF 标准公式与 `k` 含义，并准备 1 个调参或冲突案例。
